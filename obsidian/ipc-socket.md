# IPC socket

- The IPC socket is a unix-domain socket that exposes the compositor's state to external programs.
- Its path is exported in the `$KALIN_IPC_SOCKET` environment variable, set before [[kalin-wm]] forks its startup command (so children inherit it).
- It is implemented in the `ipc.c` runtime module.

- The socket broadcasts [[viewport]]/compositor state as newline-delimited JSON (camera x/y/zoom, [[follow-mode]] flags, [[crop-mode]] active, focused window).
- It accepts camera commands from clients: `pan`, `zoom`, `zoom-reset`, `follow-toggle`.
- It also accepts window-docking commands: `dockprep <appid> <x> <y> <w> <h>` arms a one-shot "dock this app_id straight into this rect the moment it maps" request (`dockprep_register()`/`dockprep_consume()` in `dwl.c`) — send *before* spawning a panel's backing terminal, since `dock` itself would no-op on an app_id that doesn't exist as a client yet. Makes a panel's first-ever spawn dock straight from its first frame: no flash at a default floating position, no camera jump chasing it there. `dock <appid> <x> <y> <w> <h>` pins a client (found by app_id) into an exact borderless screen rect, exempt from camera pan/zoom; `undock <appid>` releases it. See `setdocked()` in `dwl.c` — lets a shell panel embed a real, fully interactive client (not a rendered texture) at a fixed spot in its own QML layout. The first time a client is ever docked, `setdocked()` permanently tags it `c->ispanel` — unlike `c->docked` (transient — false again during the brief undock-before-minimize window on close), this never clears, and is the actual compositor-wide "this is shell-panel chrome, not a real window" check: excluded from the taskbar (never gets a foreign-toplevel handle), camera-follow, `Super+0`/`Super+O` fit-all, `Super+Tab` cycling, and `Super`+arrow directional focus.
- `minimize <appid> <0|1>` hides/shows a client by app_id (wraps `setminimized()`) — pairs with dock/undock so a docked panel can fully disappear on close and reappear already-running on reopen.
- The state broadcast also includes `"dock_hover":"<appid>"|null` — the app_id of whichever docked client the cursor currently sits over, or `null`. Updated in `motionnotify()` (`dwl.c`) alongside the existing `xytonode()` hit-test, only broadcasting on an actual enter/leave transition. Exists because a docked client is a real Wayland toplevel, not QML content — the shell has no `HoverHandler`-equivalent way to know the cursor is over it on its own; the compositor is the only thing that knows. Consumed by `KalinViewport.dockHoverAppId` in the shell to auto-close a docked panel when the cursor leaves it (see [[quickshell-shell]]'s `DockedPanel.qml` section).
- **Output/display settings**: the state broadcast includes `"outputs":[{"name","x","y","width","height","refresh","scale","enabled","modes":[{"width","height","refresh"},...]},...]` — every connected monitor's current state and full advertised mode list, built in `ipc_build_state()` from the `mons` list directly (no `wlr-randr` subprocess needed). `set-output <name> <w> <h> <refresh> <scale> <x> <y> <enabled>` reconfigures one by output name via `ipc_set_output()` (`dwl.c`) — the same underlying `wlr_output_state`/commit path `outputmgrapplyortest()` uses for an external wlr-output-management-v1 client (that path, inherited from dwl, already fully implements apply/test — confirmed working by hand, not something this project had to build), just addressed by name instead of a client-supplied config object. `<w>`/`<h>` `<=0` leaves the mode unchanged, `<scale><=0` leaves scale unchanged, so a caller repositioning or disabling an output doesn't need to already know its current mode/scale.
- **Screenshots** (2026-07-15): `screenshot-ui` opens the interactive [[screenshot-ui]] (same as Super+Shift+S), `screenshot` fires an immediate whole-monitor capture (same as Super+Print, lands in `$KALIN_SHOT_DIR` or `$HOME`) — for shell widgets/scripts, and for driving a nested test instance where key injection isn't possible.
- **Brightness**: the state broadcast includes `"brightness":{"value":<raw>,"max":<raw>}|null` (raw sysfs-style values, not percent; `null` if no backlight device exists). `set-brightness <value>` sets it. Implemented in `code/src/modules/backlight.c` via **systemd-logind's `SetBrightness` D-Bus method**, not a direct sysfs write — the backlight brightness file is root-owned with no group-write udev rule on the reference host, so a plain-user sysfs write fails; logind already runs as root and exposes exactly this as an authenticated per-session action. Requires the calling process to genuinely belong to an active foreground logind session (falls back from `GetSessionByPID` to scanning `ListSessions` for a real seated session if the caller itself isn't tracked by one, but `SetBrightness` itself still checks the *caller's* session/credentials — no session-path argument can substitute for that).

- The state broadcast includes `"overview":true|false` — mirrors [[overview-mode]]'s `overview_is_active()`. Added so the shell can show the connection-graph lines whenever overview is open, not just while Super is held (the two conditions used to be conflated; see the [[ledger]]).

- The [[quickshell-shell]] uses the socket (via its `KalinViewport` service) to mirror the camera and drive it.
- The socket complements [[foreign-toplevel]], which carries the window list; together they are the shell-integration channel.
